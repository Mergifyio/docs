---
title: Migrate from Partition Rules to Scopes
description: Step-by-step guide to replace the deprecated partition_rules with scopes and scope-aware CI.
---

`partition_rules` are deprecated and will be removed in a future release. This
guide walks you through replacing them with **scopes** — a more powerful system
that integrates with your CI and unlocks smarter batching, parallel merge
queues, and monorepo-aware workflows.

## Why Migrate?

Partition rules split the merge queue into independent lanes based on file
conditions evaluated by Mergify. Scopes improve on this in several ways:

| | Partition Rules | Scopes |
|---|---|---|
| **CI awareness** | None — CI runs everything | Jobs can be skipped when their scope is not affected |
| **Build tool support** | None | Native support for Nx, Bazel, Turborepo, and custom tooling |
| **Batching** | Fixed lanes — PRs routed to one partition | Overlap-aware — PRs sharing scopes batch together |
| **Parallel mode** | Not compatible | Fully supported — non-overlapping scopes merge simultaneously |
| **Merge queue scope** | Not available | Special `merge-queue` scope lets you gate expensive tests to queue-only runs |

In short, scopes give you **faster CI, smarter batching, and less wasted
compute**.

## Requirements

### GitHub Actions

You need the [`gha-mergify-ci`](https://github.com/Mergifyio/gha-mergify-ci)
GitHub Action to detect scopes and report CI status. It handles:

- Reading your scope definitions from `.mergify.yml`
- Comparing the pull request diff against file patterns
- Exposing scope outputs so downstream jobs can be conditionally skipped
- Aggregating job results so required checks always report

If you use a monorepo build tool, `gha-mergify-ci` also provides git refs for
accurate affected-project detection:

| Build Tool | How scopes are detected |
|---|---|
| **File patterns** | `gha-mergify-ci` matches changed files against patterns in `.mergify.yml` |
| **Nx** | `nx show projects --affected` using refs from `gha-mergify-ci` |
| **Turborepo** | `turbo run build --dry=json --filter` using refs from `gha-mergify-ci` |
| **Bazel** | `bazel query` for affected packages using refs from `gha-mergify-ci` |

See the [Scopes reference](/merge-queue/scopes) for detailed setup guides for
each build tool.

## Migration Steps

### 1. Map partition conditions to scope file patterns

Each partition condition becomes a scope with file patterns. The syntax changes
from Mergify conditions to glob patterns.

**Before — partition_rules:**

```yaml
partition_rules:
  - name: backend
    conditions:
      - files~=^services/api/
  - name: frontend
    conditions:
      - files~=^apps/web/
  - name: docs
    conditions:
      - files~=^docs/
  - name: default
    fallback_partition: true

queue_rules:
  - name: default
    batch_size: 5
    queue_conditions:
      - check-success = ci

pull_request_rules:
  - name: Queue PRs for merge
    conditions:
      - base = main
      - label = ready
    actions:
      queue:
```

**After — scopes with file patterns:**

```yaml
scopes:
  source:
    files:
      backend:
        include:
          - services/api/**/*
      frontend:
        include:
          - apps/web/**/*
      docs:
        include:
          - docs/**/*

queue_rules:
  - name: default
    batch_size: 5

pull_request_rules:
  - name: Queue PRs for merge
    conditions:
      - base = main
      - label = ready
    actions:
      queue:
```

:::note
  The fallback partition has no equivalent in scopes. If a PR matches no scope,
  it is treated as having no scope and will be batched with other unscoped PRs.
  This is usually the desired behavior — files outside any scope boundary don't
  need special routing.
:::

#### Pattern syntax differences

Partition rules use **regex** conditions (`files~=^pattern/`), while scopes use
**glob** patterns. Here are common conversions:

| Partition condition | Scope pattern |
|---|---|
| `files~=^services/api/` | `services/api/**/*` |
| `files~=^apps/web/.*\.(js\|ts)$` | `apps/web/**/*.{js,ts}` |
| `files~=^(config\|docker)/` | Two includes: `config/**/*` and `docker/**/*` |

### 2. Update your GitHub Actions workflow

Add a job that detects and submits scopes on every pull request using
[`gha-mergify-ci`](https://github.com/Mergifyio/gha-mergify-ci):

```yaml
name: Scopes
on:
  pull_request:

jobs:
  detect-scopes:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v5

      - name: Detect and submit scopes
        uses: Mergifyio/gha-mergify-ci@v11
        with:
          action: scopes
          token: ${{ secrets.MERGIFY_TOKEN }}
```

:::tip
  Did you know? Scopes also work outside of the merge queue to optimize your
  CI pipelines. See the [GitHub Actions integration guide](/monorepo-ci/github-actions)
  for a complete setup walkthrough.
:::

### 3. Remove partition_rules from your config

Delete the entire `partition_rules` section from `.mergify.yml`. If you
referenced `partition-name` or `current-partition-name` in your
`pull_request_rules` conditions, remove those conditions — they no longer apply.

### 4. Enable parallel mode

With partitions, independent lanes ran in parallel by design. With scopes, you
get the same behavior by enabling parallel mode:

```yaml
merge_queue:
  mode: parallel
  max_parallel_checks: 5

scopes:
  source:
    files:
      backend:
        include:
          - services/api/**/*
      frontend:
        include:
          - apps/web/**/*
      docs:
        include:
          - docs/**/*

queue_rules:
  - name: default
    batch_size: 3
```

In parallel mode, PRs touching different scopes are tested and merged
simultaneously — just like partition lanes, but with smarter dependency tracking
when scopes overlap.

See [Parallel Scopes](/merge-queue/parallel-scopes) for details.

## Using a Build Tool Instead of File Patterns

If your monorepo uses Nx, Turborepo, or Bazel, you can let the build tool
determine affected scopes instead of maintaining file patterns manually.

Switch from `files` to `manual` source:

```yaml
scopes:
  source:
    manual:
```

Then update your workflow to compute and upload scopes. Here is an example with
Nx:

```yaml
detect-scopes:
  runs-on: ubuntu-24.04
  steps:
    - uses: actions/checkout@v5
      with:
        fetch-depth: 0

    - name: Get git refs
      id: refs
      uses: Mergifyio/gha-mergify-ci@v11
      with:
        action: scopes-git-refs

    - name: Detect affected projects
      id: affected
      run: |
        scopes=$( npx nx show projects \
          --affected \
          --base=${{ steps.refs.outputs.base }} \
          --head=${{ steps.refs.outputs.head }} | paste -sd, -)
        echo "scopes=$scopes" >> "$GITHUB_OUTPUT"

    - name: Upload scopes
      uses: Mergifyio/gha-mergify-ci@v11
      with:
        action: scopes-upload
        scopes: ${{ steps.affected.outputs.scopes }}
        token: ${{ secrets.MERGIFY_TOKEN }}
```

See the dedicated guides for [Nx](/merge-queue/scopes/nx),
[Turborepo](/merge-queue/scopes/turborepo), and
[Bazel](/merge-queue/scopes/bazel).

## Complete Before & After Example

### Before: partition_rules with full CI on every partition

**.mergify.yml:**
```yaml
partition_rules:
  - name: backend
    conditions:
      - files~=^services/api/
  - name: frontend
    conditions:
      - files~=^apps/web/
  - name: shared
    fallback_partition: true

queue_rules:
  - name: default
    batch_size: 5

pull_request_rules:
  - name: Queue PRs for merge
    conditions:
      - base = main
      - "#approved-reviews-by >= 1"
    actions:
      queue:
```

**.github/workflows/ci.yml:**
```yaml
name: CI
on:
  pull_request:
  merge_group:

jobs:
  backend-tests:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v5
      - run: cd services/api && make test

  frontend-tests:
    runs-on: ubuntu-24.04
    steps:
      - uses: actions/checkout@v5
      - run: cd apps/web && npm test

  all-tests:
    needs: [backend-tests, frontend-tests]
    runs-on: ubuntu-24.04
    steps:
      - run: echo "All tests passed"
```

### After: scopes with parallel mode

**.mergify.yml:**
```yaml
scopes:
  source:
    files:
      backend:
        include:
          - services/api/**/*
      frontend:
        include:
          - apps/web/**/*

merge_queue:
  mode: parallel
  max_parallel_checks: 5

queue_rules:
  - name: default
    batch_size: 5

pull_request_rules:
  - name: Queue PRs for merge
    conditions:
      - base = main
      - "#approved-reviews-by >= 1"
    actions:
      queue:
```

The key improvements:
- **Scopes enable smarter batching** — PRs touching the same scope are grouped together
- **Non-overlapping scopes merge simultaneously** — just like partition lanes, but with smarter dependency tracking when scopes overlap
